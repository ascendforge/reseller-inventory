## Architecture

The app stays a Tauri 2 desktop app with a single-file vanilla HTML/JS frontend
(`src/index.html`, no bundler). New capability is added via official Tauri v2 plugins and
configuration; no framework or build step is introduced. Distribution moves from the
existing macOS-only `build-mac.yml` to a tag-triggered cross-platform release workflow.

### Unit 1 â€” CSV export (native Save As)

**Purpose:** Let the user export the inventory to a `.csv` file they choose, from the
packaged desktop app.

**Verification first (YAGNI):** Build/run the packaged app and click **Export CSV**. If a
file is saved correctly, *stop* â€” no code change needed beyond documenting it. Only if the
download does not fire (expected in WebView2) proceed to the change below.

**Change (if needed):**
- Add plugins: `tauri-plugin-dialog`, `tauri-plugin-fs` (Rust deps + register in `lib.rs`).
- Frontend `exportCsv()` (in `src/index.html`) becomes async with a branch:
  - **Under Tauri** (`window.__TAURI__` present): open a save dialog
    (`save({ defaultPath: 'inventory-YYYY-MM-DD.csv', filters: [{ name: 'CSV', extensions: ['csv'] }] })`);
    if the user picks a path, `writeTextFile(path, csv)` and toast success; if they cancel, do nothing.
  - **Browser fallback** (no Tauri): keep the existing `<a download>` data-URL behavior so
    the standalone-HTML mode still works.
- CSV content/columns are unchanged from the current implementation.

**Capabilities:** add `dialog:default` and `fs:allow-write-text-file` to
`src-tauri/capabilities/default.json`.

**Dependencies:** `tauri-plugin-dialog`, `tauri-plugin-fs`, the existing `state.items` data.

### Unit 2 â€” Auto-updater runtime (in-app)

**Purpose:** On launch (and on demand), detect a newer published release, and let the user
download + install it, then relaunch.

**Plugins / registration:**
- `tauri-plugin-updater` (desktop-only target dependency) and `tauri-plugin-process`
  (for relaunch), registered in `src-tauri/src/lib.rs`.

**Configuration (`src-tauri/tauri.conf.json`):**
- `plugins.updater.endpoints = ["https://github.com/ascendforge/reseller-inventory/releases/latest/download/latest.json"]`
- `plugins.updater.pubkey = "<public key generated by tauri signer>"`
- `bundle.createUpdaterArtifacts = true` (required in Tauri v2 to emit update artifacts + signatures)
- `bundle.targets`: include `nsis` (Windows) alongside `app` + `dmg` (macOS)
- `bundle.icon`: add `"icons/icon.ico"` (prerequisite Windows fix)

**Capabilities:** add `updater:default` and `process:allow-restart` to
`src-tauri/capabilities/default.json`.

**Frontend flow (`src/index.html`):**
- After storage init on launch, call `check()` from `@tauri-apps/plugin-updater`
  (accessed via the global Tauri API, consistent with `withGlobalTauri: true`).
- If an update is returned: show a small in-app prompt styled like the existing toast/overlay â€”
  *"Version X.Y.Z available â€” Install now / Later"*.
- **Install now** â†’ `update.downloadAndInstall()` with a progress indicator â†’ `relaunch()`.
- **Later** â†’ dismiss; the check repeats next launch.
- Add a manual **"Check for updates"** link in the header that runs the same flow and, when
  already up to date, toasts "You're on the latest version."
- All updater calls are wrapped so that being offline or running in browser-fallback mode
  fails silently (no error toast for the absence of Tauri).

**Dependencies:** `tauri-plugin-updater`, `tauri-plugin-process`, the published `latest.json`.

### Unit 3 â€” Signing key

**Purpose:** The updater requires every update artifact to be signed; the app verifies the
signature against an embedded public key.

- Generate a keypair with `tauri signer generate` (stored locally outside the repo, e.g.
  `~/.tauri/reseller-inventory.key`; **private key never committed**).
- Public key â†’ `plugins.updater.pubkey` in `tauri.conf.json`.
- Private key + password â†’ GitHub **repo secrets** `TAURI_SIGNING_PRIVATE_KEY` and
  `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` (set via `gh secret set`).

### Unit 4 â€” CI / release workflow

**Purpose:** On a version tag, build signed installers for both platforms and publish a
single GitHub Release with the updater manifest.

- Replace `.github/workflows/build-mac.yml` with `.github/workflows/release.yml`.
- Trigger: `push` of tags matching `v*`.
- Matrix: `macos-latest` (universal â€” `--target universal-apple-darwin`) and `windows-latest`.
- Steps per job: checkout â†’ setup Node â†’ setup Rust (with the macOS universal targets) â†’
  `npm install` â†’ `tauri-apps/tauri-action@v0` with:
  - `tagName: ${{ github.ref_name }}`, `releaseName`, `releaseDraft: false`
  - `uploadUpdaterJson: true`, `updaterJsonPreferNsis: true`, `uploadUpdaterSignatures: true`
  - env: `GITHUB_TOKEN`, `TAURI_SIGNING_PRIVATE_KEY`, `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`
- Output: one Release per tag containing the Windows NSIS installer, the macOS `.dmg`/`.app`
  + `.app.tar.gz`, their `.sig` files, and `latest.json`.

### Unit 5 â€” Repo bootstrap

**Purpose:** Put the project under version control and on GitHub so releases can be published.

- `git init` in the project root; confirm `.gitignore` covers `node_modules/`,
  `src-tauri/target/`, `src-tauri/gen/` (it does).
- Create **public** `ascendforge/reseller-inventory` under the `ascendforge` GitHub account
  (switch with `gh auth switch` as needed), push the initial commit.

## Release flow (everyday usage)

1. Bump `version` in `src-tauri/tauri.conf.json`.
2. Commit.
3. `git tag vX.Y.Z && git push --tags`.
4. CI builds both platforms, signs, and publishes the Release + `latest.json`.
5. Installed apps detect the newer version on next launch (or manual check) and offer to update.

## Error handling

- **Save dialog cancelled** â†’ no-op, no error.
- **CSV write fails** â†’ existing "Could not save â€” try again" toast pattern.
- **Update check offline / no Tauri** â†’ silent; manual check surfaces a neutral message.
- **Update download/install fails** â†’ toast with a retry affordance; app continues running
  on the current version.
- **Signature mismatch** â†’ updater refuses the artifact (built-in); surfaced as an install failure.

## Testing / verification

- **CSV:** run the packaged app, export, confirm a `.csv` is written to the chosen path with
  correct rows; confirm the browser-fallback path still downloads in standalone mode.
- **Updater (end-to-end):** publish `vCURRENT`, install it; bump to `vCURRENT+1`, tag/publish;
  launch the installed `vCURRENT` and confirm it detects, downloads, installs, and relaunches
  into the new version. Confirm the manual "Check for updates" button reports up-to-date when
  on the latest.
- **CI:** confirm a tag push produces a Release with installers, `.sig` files, and `latest.json`
  for both platforms.

## Out of scope

- Changing the local-save location or format.
- OS code signing / notarization (accepted unsigned).
- Linux builds.
- Auto-download without user consent (updates are user-initiated via the prompt/button).
